**背景**：修复 **issue #282** — DeepSeek `deepseek-chat` 等非多模态模型只接受 `content` 为字符串，旧实现无条件输出 `[{"type":"text","text":...}]` 导致 `invalid_request_error`。


### 测试用例

| 测试类 | 测试方法 | 输入 | 期望 |
|--------|----------|------|------|
| `TestCreateMessagesContentFormat` | `test_no_images_emits_string_content` | segments=[], 无图片 | `content` 为 `str` 类型，值为 `"PROMPT_BODY"` |
| | `test_empty_image_list_emits_string_content` | segments=[], `video_img_urls=[]` | `content` 为 `str` 类型（空列表也走纯文本分支） |
| | `test_with_images_emits_multimodal_array` | segments=[], 2 张图片 URL | `content` 为 `list`，含 1 text + 2 image_url，detail="auto" |

| | `test_merge_messages_no_image_url_field` | 1 个 partial | JSON 序列化后不含 `image_url` |

### 覆盖的核心逻辑

- ✅ **无图片 → string content**（非多模态模型兼容）
- ✅ **有图片 → array content**（多模态模型功能不退化）

## 九、技术特征总结

### 9.1 模块加载方式

所有 Backend 单元测试均使用 **`importlib.util.spec_from_file_location`** 动态加载被测模块，而非通过包导入：

```python
ROOT = pathlib.Path(__file__).resolve().parents[1]
MODULE_PATH = ROOT / "app" / "gpt" / "request_chunker.py"
spec = importlib.util.spec_from_file_location("request_chunker", MODULE_PATH)
module = importlib.util.module_from_spec(spec)
spec.loader.exec_module(module)
```

**优点**：绕过 `__init__.py` 的级联导入链，避免触发数据库初始化、环境变量依赖等副作用。

### 9.2 依赖隔离

| 测试文件 | Stub 的依赖 |
|----------|------------|
| `test_request_chunker.py` | 无外部依赖（纯逻辑） |
| `test_note_helper.py` | 无外部依赖（纯逻辑） |
| `test_screenshot_marker.py` | 无外部依赖（纯逻辑） |
| `test_universal_gpt_checkpoint.py` | `app.gpt.base`, `app.gpt.prompt_builder`, `app.gpt.prompt`, `app.gpt.utils`, `app.gpt.request_chunker`, `app.models.*` |
| `test_universal_gpt_content_format.py` | 同上 |
| `test_video_reader_dedupe.py` | `app.utils.logger`, `app.utils.path_helper`, `PIL`, `ffmpeg`, `subprocess` |

### 9.3 测试数据设计

- **轻量 dataclass**：使用 `DummySeg`、`_FailingCompletions` 等替代真实模型
- **基于字节大小的分块**：`size_estimator` 按字节估算（非 token），快速可预测
- **二进制帧模拟**：通过 `b"frame-a"` / `b"frame-b"` 区分帧内容

## 十二、已知局限性

1. **覆盖率集中在工具类**：GPT 模块（chunker, universal_gpt）占 12/19 个测试方法，其他模块覆盖较少
2. **前端主项目无测试**：`BillNote_frontend/` 完全没有单元测试
3. **Extension 测试薄弱**：仅 1 个组件渲染测试 + 1 个冒烟测试
4. **纯逻辑测试**：所有测试都不涉及真实 API 调用或文件 I/O，无法覆盖网络错误、磁盘满等场景
5. **无参数化测试**：缺少边界值（空字符串、超长文本、特殊字符）的参数化覆盖
