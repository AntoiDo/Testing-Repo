## 三、测试基础设施

### 3.1 数据库引擎隔离

**问题**：原 `app/db/engine.py` 在模块 import 时立即创建 engine 单例，多个测试文件设置不同的 `DATABASE_URL` 环境变量时，只有第一个加载的模块的 URL 生效，后续模块的数据库操作全部指向错误的临时目录。

**解决方案**：

| 组件 | 改动 | 说明 |
|------|------|------|
| `app/db/engine.py` | 懒加载 + URL 变更检测 | `get_engine()` 首次调用时才创建 engine；若 `DATABASE_URL` 变更，自动 dispose 旧 engine 并以新 URL 重建 |
| `tests/conftest.py` | 模块级 engine 重置 | 每个测试模块运行前通过 `sys.modules` 直接操作 engine 全局变量重置，确保下一模块使用正确的 `DATABASE_URL` 重建 |
| `.github/workflows/ci.yml` | 添加 `working-directory: backend` | 确保 pytest 在后端目录内运行，正确加载 conftest.py 和测试模块 |

### 3.2 跨文件隔离改进（2026-06-13）

| 问题 | 修复 |
|------|------|
| GPT 单元测试文件在模块级向 `sys.modules` 注入空壳 `GPT`/`GPTSource`，污染 pipeline 集成测试的 import | 将 stub 注入移至 `setUpClass`/`setUpModule`，在 `tearDownClass`/`tearDownModule` 中还原 `sys.modules` |
| 数据库/流水线测试文件在模块级用空壳覆盖 `fastapi`，导致 API 测试异常处理器找不到 `Request` | 移除 `fastapi` stub（真实 fastapi 已在 CI 和本地完整安装），仅保留 `app.routers.*` stub 阻断不必要的下载器导入链 |
| 三个集成测试文件模块级 `DATABASE_URL` 环境变量按字母序互相覆盖 | 每个文件添加 `setUpModule`，在测试运行前恢复自身的 `DATABASE_URL` |

### 3.3 隔离策略

| 文件 | 数据库 | 清理方式 |
|------|--------|----------|
| `test_integration_pipeline.py` | 临时 SQLite（`tempfile.mkdtemp`） | `tearDownModule` 删除整个临时目录 |

- **重型依赖 stub**：`ffmpeg`、`app.routers.*` 使用运行时模块替换（`sys.modules`），避免触发下载器/服务层完整导入链
- **轻量依赖直接导入**：`app.db.*`、`app.validators.*`、`app.models.*` 等直接导入使用

## 九、测试可靠性保障（2026-06-13 完善）

| 保障项 | 说明 |
|--------|------|
| **数据库懒加载** | engine 不再在 import 时创建，首次 `get_engine()` 调用时才初始化 |

| **stub 生命周期管理** | stub 安装/清理限定在 setUp/tearDown 生命周期内，不泄漏到其他测试文件 |

## 十、已知局限性

1. **无性能测试**：未覆盖并发请求、大文件下载、长视频转写等场景
2. **无真实外部依赖**：LLM API、youtube-dl、Whisper 均为 stub，无法验证真实网络交互
3. **仅后端覆盖**：前端 `BillNote_frontend/` 无集成测试，Extension 仅有基础 E2E
4. **Python 版本要求**：源码使用 `str | None` 等 Python 3.10+ 语法，需 Python ≥ 3.10（CI 使用 3.12）
5. **transcriber 依赖**：当 `faster-whisper` 不可用时，部分 transcriber 测试会被 `skipTest`
